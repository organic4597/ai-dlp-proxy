## 2. context_multiplier 0-hit 보정값 수정

### 배경
단독 전화번호 입력 시 마스킹이 안 되는 버그를 발견했다. 로그를 보니 `confidence 0.4 < threshold 0.5`로 기록돼 있었다.

### 핵심 내용
- `_context_multiplier()`가 문맥 키워드 0개일 때 `0.4`를 반환
- `confidence = base_score × 0.4 = 0.4` → 임계값(0.5) 미달로 마스킹 스킵

### 문제 & 해결
- 증상: 전화번호 단독 입력 시 탐지는 되지만 마스킹 미적용
- 원인: 문맥 키워드 없을 때 multiplier가 너무 낮아 신뢰도가 threshold 미달
- 해결: 0-hit 반환값 `0.4 → 0.6`으로 상향 (`src/engine/pipeline/regex_stage.py`)

```python
# 수정 전
if hit_count == 0:
    return 0.4

# 수정 후
if hit_count == 0:
    return 0.6
```

## 3. TUI 성능 최적화

### 배경
트래픽 상세 화면(`_show_turn_detail`) 새로고침 시 로딩이 매우 느렸다. 진단 결과 평균 123개 messages × 34건 = 7.4 MB를 매번 렌더링하고 있었다.

### 핵심 내용
**① JSONL 저장 최적화 (`inspect_traffic.py`)**
- 전체 히스토리 저장 → last_assistant 이후 신규 메시지만 저장
- `_MSG_MAX`: 2000자 → 500자

```python
def _new_only(raw_msgs: list) -> list:
    """last_assistant 이후 메시지만 반환. 없으면 user만."""
    last_asst = -1
    for i, m in enumerate(raw_msgs):
        if m.get("role") in ("assistant", "model"):
            last_asst = i
    if last_asst >= 0:
        return raw_msgs[last_asst + 1:]
    return [m for m in raw_msgs if m.get("role") not in ("system", "developer")]
```

**② TUI 탐지 정보 탭 (`tui.py`)**
- "대상 전체 내용" 섹션 제거 — targets 전체 루프가 렌더 병목이었음

**③ TUI 전송 내용 탭 (`tui.py`)**
- 탐지된 필드만 원문 → 마스킹 결과 형태로 표시
- 탐지 없으면 새 user 메시지 500자만 표시

## 4. iptables nf_tables 문법 오류 수정

### 배경
서비스 시작 시 아래 경고가 출력되며 OUTPUT REDIRECT 규칙이 적용되지 않고 있었다. 규칙이 없으면 VS Code(ubuntu UID=1000) 트래픽이 프록시를 우회한다.

```
iptables v1.8.10 (nf_tables): unexpected ! flag before --match
[WARN] OUTPUT REDIRECT 실패 — xt_owner 모듈 확인 필요
```

### 핵심 내용
- `iptables v1.8.10 (nf_tables)` 환경에서는 negation `!` 위치가 다름
- `setup_transparent.sh` 2군데 수정

```bash
# legacy 문법 (nf_tables에서 실패)
iptables -t nat -A OUTPUT -p tcp --dport 443 ! -m owner --uid-owner 0 -j REDIRECT --to-ports 4443

# nf_tables 문법 (수정)
iptables -t nat -A OUTPUT -p tcp --dport 443 -m owner ! --uid-owner 0 -j REDIRECT --to-ports 4443
```

### 문제 & 해결
- 증상: `[WARN] OUTPUT REDIRECT 실패` — VS Code Copilot 트래픽이 프록시 우회
- 원인: `! -m owner` 문법은 iptables-legacy 전용, nf_tables는 `-m owner !` 사용
- 해결: `setup_transparent.sh` 수정 후 즉시 재적용, 규칙 확인

```bash
$ sudo iptables -t nat -L OUTPUT --line-numbers | grep REDIRECT
4  REDIRECT  tcp  --  anywhere  anywhere  tcp dpt:https ! owner UID match root redir ports 4443
```

## 7. 공급자별 API 파서 파일 분리 (리팩토링)

### 배경

### 핵심 내용

```
src/engine/api/
├── openai.py     — parse() + summarize() + extract_messages()
├── anthropic.py  — parse() + summarize() + extract_messages()
├── gemini.py     — parse() + summarize() + extract_messages()
└── base.py       — DLPTarget, ParsedRequest (공통 타입)

src/engine/extractor.py
├── extract()             — 공급자 감지 + parse() 라우팅 (기존)
├── summarize_request()   — 공급자별 summarize() 라우팅 (신규)
└── extract_messages()    — 공급자별 extract_messages() 라우팅 (신규)
```

`inspect_traffic.py`의 인라인 분기 → 위임 1줄로 대체:

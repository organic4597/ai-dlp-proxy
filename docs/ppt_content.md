## [슬라이드 2] 배경 및 문제 정의

### AI 시대의 새로운 개인정보 유출 경로

**기존 위협**: 해킹, 내부자 유출, DB 탈취
**새로운 위협**: 사용자가 *자발적으로* LLM에 민감정보 입력

```
사용자 → AI 에이전트 → [주민번호 020102-2717299 포함된 문서 요약 요청]
                       ↓
                  OpenAI 서버 (해외)
```

**실제 우려 사례**
- 직원이 Claude에 고객 개인정보가 포함된 계약서 붙여넣기
- 개발자가 ChatGPT에 DB 쿼리 결과(개인정보 포함) 분석 요청
- 의료진이 진단 내용·환자 정보를 LLM에 전송

**핵심 문제**
- 사용자는 민감정보인지 인식 못 하거나 무시
- 기업 보안팀은 HTTPS 암호화로 내용 확인 불가
- LLM 사업자 서버에 PII가 학습 데이터로 흡수될 위험

## [슬라이드 3] 해결 아이디어

### 투명한 인라인 DLP 프록시

```
Before:  PC → LLM API  (내부 검열 없음)

After:   PC → [DLP Proxy] → LLM API
                  ↓
           PII 자동 탐지·마스킹
```

**핵심 설계 원칙**
1. **투명성**: 사용자·에이전트 코드 변경 없음 (프록시 설정만)
2. **인라인**: 요청 가로채기 → 마스킹 → 전달 (실시간 처리)
3. **불가역성**: 마스킹된 데이터는 LLM 서버에 도달하지 않음
4. **제어 가능**: TUI로 규칙 ON/OFF, 정책 실시간 변경

## [슬라이드 4] 시스템 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│                     Windows PC                               │
│  AI Agent / 브라우저                                         │
│  시스템 프록시: 192.168.0.16:4001                            │
└────────────────────────┬─────────────────────────────────────┘
                         │ HTTPS (암호화 해제)
┌────────────────────────▼─────────────────────────────────────┐
│                  Raspberry Pi (192.168.0.16)                  │
│                                                              │
│  ① mitmproxy :4001  ──→  inspect_traffic.py (addon)         │
│     · CA 인증서로 HTTPS 복호화                               │
│     · API 파서: OpenAI / Anthropic / Gemini                  │
│     · 텍스트 추출 → ②에 스캔 요청                           │
│                                                              │
│  ② engine_server.py (UDS /tmp/dlp-engine.sock)              │
│     · NDJSON 프로토콜                                        │
│     · DLP Pipeline: RegexStage                               │
│       12개 규칙, 체크섬/Luhn 검증                            │
│                                                              │
│  ③ tui.py (Textual TUI)                                     │
│     · 6탭 실시간 모니터링                                    │
│     · 규칙 ON/OFF, 정책 제어                                 │
└────────────────────────┬─────────────────────────────────────┘
                         │ HTTPS (마스킹된 요청)
┌────────────────────────▼─────────────────────────────────────┐
│              외부 LLM API                                    │
│   OpenAI / Anthropic / Google Gemini                         │
│   → PII가 제거된 안전한 요청만 수신                          │
└──────────────────────────────────────────────────────────────┘
```

**처리 흐름 요약**
1. PC의 HTTPS 요청 → mitmproxy가 CA 인증서로 복호화
2. LLM API 파서가 메시지 텍스트 추출
3. DLP 엔진이 정규식 + 알고리즘 검증으로 PII 탐지
4. 탐지 시: 마스킹(`[주민등록번호]`) 후 재암호화 → LLM 전달
5. TUI에서 실시간 확인 및 제어

## [슬라이드 5] 구현 현황 — DLP 탐지 규칙

### 12개 탐지 규칙 (Regex + 알고리즘 검증)

| 규칙 | 대상 | 등급 | 검증 방식 |
|------|------|------|-----------|
| kr_rrn | 주민등록번호 | CRITICAL | mod-11 체크섬 + 생년월일 + 성별코드 |
| credit_card | 신용카드번호 | CRITICAL | Luhn 알고리즘 |
| us_ssn | 미국 SSN | CRITICAL | 패턴 |
| aws_access_key | AWS 액세스키 | CRITICAL | `AKIA` 접두어 |
| pem_private_key | PEM 개인키 | CRITICAL | BEGIN/END 블록 |
| github_pat | GitHub 토큰 | CRITICAL | `ghp_` 접두어 |
| kr_passport | 여권번호 | HIGH | 패턴 |
| kr_driver_license | 운전면허번호 | HIGH | 패턴 |
| jwt_token | JWT 토큰 | HIGH | 3-part base64 |
| api_key_assignment | API 키 할당문 | HIGH | 컨텍스트 |
| kr_phone | 휴대전화번호 | MEDIUM | 010/011/016 패턴 |
| email | 이메일 주소 | LOW | RFC 패턴 |

**Severity → Action**
- CRITICAL → 마스킹(MASK)
- HIGH/MEDIUM/LOW → 경고(ALERT) — 정책에 따라 차단 가능

## [슬라이드 6] 구현 현황 — 마스킹 파이프라인

### 실제 동작 예시

**Before (사용자가 전송하려던 내용)**
```json
{
  "messages": [{
    "role": "user",
    "content": "이 환자 정보를 분석해줘: 홍길동, 주민번호 900101-1234568,
                카드번호 4532-1234-5678-9012, 연락처 010-1234-5678"
  }]
}
```

**After (LLM에 실제 전달된 내용)**
```json
{
  "messages": [{
    "role": "user",
    "content": "이 환자 정보를 분석해줘: 홍길동, 주민번호 [주민등록번호],
                카드번호 [신용카드번호], 연락처 [전화번호]"
  }]
}
```

**마스킹 구현 기술**
- 탐지된 match의 offset(시작·끝 위치) 기반 정밀 치환
- **역순(reverse) 적용**: 뒤에서부터 치환 → offset drift 방지
- Content-Length 헤더 자동 재계산
- `flow.request.content` 직접 교체 → mitmproxy가 재암호화

## [슬라이드 7] 구현 현황 — TUI 대시보드

### Textual 기반 6탭 인터랙티브 TUI

```
  턴 3  요청 12  스캔 12  탐지 8  마스킹 7  │  Engine ●  mitm ●
┌──────────────────────────────────────────────────────────────┐
│ 트래픽 │ 탐지 │ 제어 │ 프로세스 │ 설정 │ 로그              │
├──────────────────────────────────────────────────────────────┤
│ [트래픽 탭]                                                  │
│ ID  시각       Provider  경로                  크기   액션   │
│ 12  20:31:42  openai    /v1/chat/completions  1.2KB  masked │
│ 11  20:30:18  gemini    /v1beta/models/...    0.8KB  masked │
│ 10  20:28:55  openai    /v1/chat/completions  0.5KB  pass   │
├──────────────────────────────────────────────────────────────┤
│ [상세보기]                    [엔진 결과]                    │
│ POST /v1/chat/completions     rule: kr_rrn                  │
│ Host: api.openai.com          severity: CRITICAL             │
│ Content-Length: 1187          confidence: 1.0                │
│                               action: MASK                   │
└──────────────────────────────────────────────────────────────┘
```

**탭별 기능**
- **트래픽**: 요청 목록 + HTTP 헤더/본문 + 엔진 결과 상세
- **탐지**: 누적 findings 목록 (규칙/심각도/매칭 텍스트)
- **제어**: 파이프라인 ON/OFF, 정책 스위치, 규칙별 활성화 토글
- **프로세스**: mitmproxy·engine_server 상태 모니터링·재시작
- **설정**: 포트, 대상 도메인 설정
- **로그**: 실시간 이벤트 스트림

## [슬라이드 8] 성능 및 오탐 개선 결과

### 처리 성능

| 측정 항목 | 수치 |
|-----------|------|
| Regex Stage 스캔 응답시간 | **< 1ms** |
| 실제 트래픽 마스킹 처리 | **< 5ms** (총 오버헤드) |
| 엔진 통신 방식 | Unix Domain Socket (네트워크 무관) |

### 오탐(False Positive) 개선

**문제**: 주민번호 패턴 매칭은 되지만 실제로는 가짜 숫자인 경우

| 항목 | 개선 전 | 개선 후 |
|------|---------|---------|
| kr_rrn 총 탐지 | 79건 | 6건 |
| 그 중 오탐 | 73건 (92%) | 0건 (0%) |
| 적용 개선 | - | 체크섬 실패 → 필터링<br>생년월일 유효성 검사<br>성별코드 유효성 검사 |

**개선 핵심**: 체크섬(mod-11) 실패 시 confidence `0.7`(탐지) → `0.0`(필터링)

## [슬라이드 10] 향후 개발 계획

### Phase 4 — SLM(소형 언어모델) 통합

**목표**: 규칙 기반으로 탐지하기 어려운 문맥적 민감정보 탐지

```
"Project Artemis의 런칭 일정은..." → 규칙에 없는 기업 기밀
"혈압약을 처방받고 있어서..."      → 의료 정보 (패턴 없음)
```

- 모델: `Qwen2.5-1.5B-Q4` 또는 `EXAONE-3.5-2.4B-Q4` (한국어 특화)
- GBNF grammar로 JSON 출력 강제
- 예상 처리시간: ~150~300ms (RPi CPU 기준)

### Phase 5 — 패키지화 및 배포

- `pip install ai-dlp-proxy` 단일 명령 설치
- CA 인증서 자동 생성·시스템 등록
- Windows / macOS / Linux 크로스플랫폼

### Phase 6 — 사용자 정의 정책 (장기)

- `settings.yaml`로 커스텀 민감 키워드 정의
- 4계층 정책: 개인 > 조직 > 도메인 > 글로벌

## 📋 평가 요약

이 계획서의 2-Pass Regex 문맥 보정 + 보호 자산 탐지 아키텍처는 **학술·산업 DLP 최신 동향과 높은 정합성**을 보이며, 실용적 효용이 충분합니다. 다만, 몇 가지 한계점과 개선 기회가 존재합니다.

## 1. 아키텍처 비교 분석 — 참고 문헌 대비

### 1-1. RECAP (arXiv 2025) vs. `ai-dlp-proxy`

| 비교 항목 | RECAP [[1]](https://arxiv.org/abs/2510.07551) | `ai-dlp-proxy` 계획 |
|---|---|---|
| **파이프라인 구조** | 3단계: Regex → LLM 문맥 분석 → 필터링 | 3단계: Regex(문맥보정) → Asset → SLM |
| **Regex 역할** | 고재현율 후보 추출 (동일) | 고재현율 후보 추출 (동일) |
| **문맥 분석** | GPT 계열 LLM 프롬프팅 (외부 API 의존) | 로컬 규칙 기반 + 2B SLM (외부 의존 없음) |
| **다국어 지원** | 13개 저자원 언어 | 한국어+영어 중심 |
| **F1 성능** | fine-tuned NER 대비 +82%, zero-shot LLM 대비 +17% | 미측정 (테스트 계획만 존재) |
| **엔티티 수** | 300+ 유형 | 12개 Regex 룰 + 사용자 정의 자산 |
| **재학습 필요** | 없음 (모듈식) | 없음 (동일 장점) |

**평가**: RECAP은 대규모 외부 LLM에 의존하여 높은 정밀도를 달성하지만, 네트워크 지연과 API 비용이 발생합니다. `ai-dlp-proxy`는 **로컬 경량 규칙 기반 문맥 보정**으로 외부 의존성 없이 실시간 프록시 환경에 적합합니다. 다만, RECAP의 3단계 엔티티 disambiguation 필터링에 비해 `ai-dlp-proxy`의 규칙 기반 보정은 **표현력이 제한적**입니다.

### 1-2. Scientific Reports (Nature, 2025) vs. `ai-dlp-proxy`

| 비교 항목 | 금융 문서 하이브리드 [[2]](https://www.nature.com/articles/s41598-025-04971-9) | `ai-dlp-proxy` 계획 |
|---|---|---|
| **정밀도** | 94.7% | 미측정 |
| **재현율** | 89.4% | 미측정 |
| **F1** | 91.1% | 미측정 |
| **ML 구성** | 커스텀 NER + BERT fine-tuning | 임베딩 유사도 + 2B SLM (fine-tuning 없음) |
| **도메인** | 금융 문서 (정형) | AI 채팅 프록시 (비정형, 코드 혼재) |
| **코드 문맥 처리** | 없음 (금융 도메인 한정) | ✅ `_is_code_context()` 명시적 처리 |
| **검증 방법** | 혼동 행렬, ROC, PR 곡선 | pytest 단위 테스트 (V1~V18) |

**평가**: 금융 문서 연구는 정형 데이터에서의 높은 수치를 보이지만, **코드가 혼재된 AI 채팅 환경은 고려하지 않습니다**. `ai-dlp-proxy`의 코드 문맥 감지(`_is_code_context`)는 이 도메인 특화 문제를 직접 해결하는 **차별화 포인트**입니다.

### 1-3. CrowdStrike 특허 (US20250005175A1) vs. `ai-dlp-proxy`

| 비교 항목 | CrowdStrike 특허 [[3]](https://patents.google.com/patent/US20250005175A1/en) | `ai-dlp-proxy` 계획 |
|---|---|---|
| **접근법** | Regex 후보 → LLM 보완 스크러빙 | Regex 후보 → 문맥 보정 → 임베딩+SLM |
| **LLM 역할** | 패턴 미탐 민감정보 추가 탐지 | SLM으로 이름/주소 등 보완 탐지 (동일 목적) |
| **Confidence 점수** | 미명시 (이진 판정 추정) | ✅ 연속 confidence (0.0~1.0) + threshold |
| **문맥 보정** | LLM 내부 처리 (블랙박스) | ✅ 투명한 규칙 기반 (코드 감지 + PII 키워드) |
| **보호 자산** | 없음 (PII/자격증명 중심) | ✅ 사용자 정의 보호 자산 (키워드+임베딩) |

**평가**: CrowdStrike 특허와 동일한 하이브리드 철학을 공유하되, `ai-dlp-proxy`는 **투명한 confidence 연산 파이프라인**과 **사용자 정의 자산**이 추가된 점에서 차별화됩니다.

### 1-4. Google Cloud DLP API vs. `ai-dlp-proxy`

| 비교 항목 | Google Cloud DLP | `ai-dlp-proxy` 계획 |
|---|---|---|
| **Likelihood 점수** | 5단계 이산값 (VERY_UNLIKELY ~ VERY_LIKELY) | 연속 confidence (0.0~1.0) |
| **문맥 분석** | 주변 키워드로 likelihood 조정 | `_context_multiplier()` (동일 메커니즘) |
| **Validator** | Luhn, 체크섬 등 내장 검증 | `_validate_kr_rrn`, `_validate_card` (동일) |
| **Suppression 규칙** | 커스텀 제외 규칙 지원 | Allowlist (정규화 비교 + TTL) |
| **배포 형태** | 클라우드 API (외부 전송 필수) | 로컬 프록시 (데이터 외부 전송 없음) |

**평가**: `ai-dlp-proxy`의 문맥 보정 메커니즘은 Google Cloud DLP의 context 기반 likelihood 조정과 **거의 동일한 아키텍처 패턴**을 따릅니다. 로컬 실행이라는 점에서 **데이터 주권(data sovereignty)** 측면에서 우위가 있습니다.

### 1-5. Microsoft Presidio vs. `ai-dlp-proxy`

| 비교 항목 | Microsoft Presidio | `ai-dlp-proxy` 계획 |
|---|---|---|
| **구조** | Regex + NER(spaCy) + Context Analyzer | Regex(문맥보정) + Asset(임베딩) + SLM |
| **Confidence** | 인식기별 점수 → 문맥 분석기가 조정 | validator/코드감지/키워드 부스트로 조정 |
| **F1 (하이브리드)** | 0.90~0.97 (엔티티별 상이) | 미측정 |
| **다국어** | 모델 의존 (spaCy 언어팩) | 한국어 키워드 직접 내장 + `re.ASCII` |
| **코드 문맥** | 없음 | ✅ 강/약 시그널 기반 코드 감지 |

## 2. 실질적 효용성 평가

### 2-1. 오탐 감소 효과 (핵심 가치)

계획서의 **연산 예시 표**를 기반으로 분석:

| 시나리오 | 현재 시스템 | 계획 적용 후 | 개선 |
|---|---|---|---|
| `import hashlib; secret_key = hashlib.sha256(data)` | conf=1.0 → **오탐 action** | conf=0.3×0.4=**0.12** → 필터링 | ✅ 오탐 제거 |
| `M12345678` (변수명, 키워드 없음) | conf=1.0 → **오탐 action** | conf=1.0×0.4=**0.4** → 필터링 | ✅ 오탐 제거 |
| `def parse(m12345678)` | conf=1.0 → **오탐 action** | conf=0.3×0.4=**0.12** → 필터링 | ✅ 오탐 제거 |
| `"제 전화번호는 010-1234-5678입니다"` | conf=1.0 → action | conf=1.0×1.3=**1.0** → action | ✅ 정탐 유지 |
| `"880515-1104333"` 코드 내 (체크섬 통과) | conf=1.0 → action | conf=**0.8** (floor) → action | ✅ 정탐 유지 |
| `"Please return this document 010-1234-5678"` | conf=1.0 → action | weak 1개만 → 코드 비활성 → conf≥1.0 | ✅ 정탐 유지 |

**효용 판정**: 현재 코드에서 `scan()`은 validator 외에는 모두 conf=1.0으로 처리하므로, 계획서의 2-Pass 보정은 **이론적으로 오탐을 ~70% 이상 감소**시킬 수 있습니다 (코드 문맥 + PII 키워드 0개 조합이 가장 빈번한 오탐 패턴).

### 2-2. 성능 영향 (레이턴시)

| 단계 | 추가 연산 | 예상 오버헤드 |
|---|---|---|
| `_is_code_context()` | 2개 정규식 findall × ±100자 윈도우 | **< 0.1ms/매치** (무시 가능) |
| `_context_multiplier()` | dict lookup + `in` 검색 (최대 10개 키워드) | **< 0.05ms/매치** (무시 가능) |
| Allowlist 비교 | 정규화 + set 순회 (최대 100개) | **< 0.1ms/매치** |
| AssetStage 키워드 | 전문 `in` 검색 | **< 1ms** |
| AssetStage 임베딩 | ONNX 추론 (ko-sroberta, CPU) | **8~12ms/청크** (계획서 추정치와 일치) |
| NMS | O(n²) 정렬+비교, n ≤ ~50 | **< 0.1ms** |

**판정**: Phase 1~2는 **성능 영향 무시 가능**. Phase 3 임베딩은 청크 수에 비례하므로, 1000자 텍스트 ≈ 5~7 청크 × 10ms = **50~70ms** 추가. 프록시 환경에서 500ms 타임아웃은 적절합니다.

## 3. 강점 분석

### ✅ 아키텍처 투명성
- RECAP·CrowdStrike 특허의 LLM 블랙박스 판정과 달리, confidence 계산이 **완전히 추적 가능**
- `코드 패널티(×0.3) → PII 키워드 배율 → Validator floor → cap(1.0)` 순서가 명시적
- 감사(audit) 요구사항 충족: 모든 finding이 리포트에 남음 (suppressed 포함)

### ✅ 코드 문맥 감지 (차별화)
- 기존 DLP 연구/제품 중 **코드와 자연어가 혼재된 AI 채팅** 시나리오를 명시적으로 처리하는 사례가 드뭄
- `_STRONG_CODE_RE` / `_WEAK_CODE_RE` 분리 + `re.ASCII`로 한글 조사 문제 해결 → 실무적 배려

### ✅ Validator 동적 Floor
- Google DLP의 likelihood 조정과 유사하되, **체크섬 종류별 신뢰도 차등** (kr_rrn 0.8 > credit_card 0.6)
- 이는 학술 문헌에서도 권장되는 "validator 강도에 따른 confidence 조정" 패턴

### ✅ 가혹 정규화(Aggressive Normalization) Allowlist
- `re.sub(r'[\W_]+', '', v).casefold()` → 공백/특수문자 삽입 우회 공격 방지
- (룰명, 정규화값) 쌍 비교로 충돌 방지
- **TTL 만료** 기능은 산업 DLP 제품(Microsoft Purview 등)에서도 권장하는 패턴

### ✅ 점진적 배포 전략
- Phase 1은 외부 의존성 없이 즉시 구현 가능
- `context_penalty_enabled: false` 롤백 플래그로 안전한 배포
- `sentence-transformers` 미설치 시 graceful fallback

## 4. 한계점 및 개선 기회

### ⚠️ 한계 1: 정량적 성능 평가 부재

**문제**: 계획서에는 V1~V18 단위 테스트만 있고, **벤치마크 데이터셋 기반 정량 평가(Precision/Recall/F1)**가 없습니다.

- RECAP은 `nervaluate` 라이브러리로 13개 언어 F1을 보고
- Scientific Reports 논문은 혼동 행렬, ROC, PR 곡선을 제시
- Presidio는 `presidio-research`에서 자체 벤치마크 도구 제공

**권장**: 
```
V19: 한국어 PII 벤치마크 데이터셋(최소 500문장, 코드 혼재 30%) 구축
     → Phase 1 적용 전후 Precision/Recall/F1 비교 리포트 생성
     → 목표: Precision +20%p 이상 (현재 추정 ~60% → 80% 이상)
```

### ⚠️ 한계 2: PII 키워드 세트의 정적 특성

`_PII_CONTEXT_WORDS`는 **수동 관리** 딕셔너리입니다. 새 도메인(의료, 법률 등) 추가 시 키워드를 직접 수정해야 합니다.

- RECAP은 LLM이 자체적으로 문맥을 판단 (키워드 목록 불필요)
- Google Cloud DLP은 ML 기반 context analyzer 사용

**권장**: 현재 설계는 12개 룰에 대해 충분하지만, 향후 룰 확장 시 **TF-IDF 기반 자동 키워드 추출** 또는 **few-shot SLM 기반 문맥 판정** 전환을 고려할 수 있습니다.

### ⚠️ 한계 3: SLM(Gemma 4 2B-IT) 한국어 PII 탐지 성능 불확실

검색 결과, Gemma 2B-IT의 **한국어 PII 탐지 전용 벤치마크는 2025년 기준 미공개**입니다. 다국어 일반 성능은 확인되나, 한국어 주민등록번호/전화번호 등의 문맥 탐지 정확도는 검증되지 않았습니다.

**권장**: Phase 3 진입 전 **한국어 PII 탐지 mini-benchmark**(100문장) 수행. Gemma 4 2B vs Qwen2.5-1.5B vs Phi-3-mini 비교 후 최종 모델 선정.

### ⚠️ 한계 4: 임베딩 유사도 임계값 고정 시작

기본 `embedding_threshold: 0.80`은 보수적 시작으로 적절하나, 자산별 최적 임계값을 찾는 **자동 캘리브레이션** 메커니즘이 없습니다.

- Presidio의 `presidio-research`는 데이터셋 기반 자동 threshold 튜닝 도구 제공

**권장**: TUI에서 "테스트 문장 입력 → 유사도 점수 실시간 확인" 기능 추가로 사용자가 직관적으로 임계값을 조정할 수 있게 하는 것이 좋습니다.

### ⚠️ 한계 5: NMS가 단순 구간 겹침만 검사

현재 NMS는 `start < end AND end > start`(IoU > 0)면 무조건 우선순위 낮은 쪽을 suppress합니다. 이는 **부분 겹침에서 두 탐지가 모두 유효한 경우**(nested entity)를 처리하지 못합니다.

- NER 분야에서는 **nested NER** 처리를 위해 IoU 임계값(0.5 등)을 두거나, span-level merging을 사용

**권장**: 현재 12개 룰 + 자산 수준에서는 단순 겹침 처리가 충분하지만, 룰 수 증가 시 IoU 임계값 도입을 검토하세요.

## 5. 산업 대비 포지셔닝

```
             정밀도
              ↑
              │
  Presidio ●  │  ● Google Cloud DLP
  (오픈소스)  │    (클라우드 SaaS)
              │
              │      ● RECAP (연구)
              │
  ai-dlp-proxy│● (예상 위치)
  (로컬 프록시)│
              │
              │  ● CrowdStrike (엔터프라이즈)
              │
  순수 Regex ●│
              └──────────────────────→ 레이턴시 (낮을수록 좋음)
```

`ai-dlp-proxy`는 **로컬 실행 + 낮은 레이턴시 + 중간~높은 정밀도** 영역에 위치합니다.
순수 Regex 대비 정밀도를 크게 끌어올리면서도, 클라우드 DLP 대비 데이터 외부 전송이 없는 것이 핵심 가치입니다.
